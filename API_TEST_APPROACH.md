# API Test Approach

This document explains how API tests are organized in `playwright-2025` so that contributors (and AI agents) can extend the suite without reverse engineering existing files. Treat it as the canonical checklist before adding or updating tests.

## Core Principles
- Keep Playwright tests readable by following the `// given`, `// when`, `// then` comment convention in every test block.
- One HTTP **method + path** per file: every endpoint has a matching client in `http/` and a matching spec in `tests/api/**`. Never mix multiple endpoints or verbs in the same file.
- Order test cases by response code (e.g. success `200/201`, validations `400`, auth failures `401`, `403`, `404`, …).
- Use data generated by `@faker-js/faker` through the helpers in `generators/` to avoid collisions and keep tests idempotent.
- Prefer a small number of validation (400) checks. Focus on happy paths and the most meaningful authorization/error flows.

## Project Layout & Shared Utilities
- `tests/api/**`: API specs, named `*.api.spec.ts`. Folder hierarchy mirrors the resource (`tests/api/products/createProduct.api.spec.ts`, etc.).
- `http/**`: thin HTTP clients wrapping `APIRequestContext`. Each request file exports functions such as `createProduct`, optional variants like `createProductWithoutAuth`, and hard-codes the API path via `API_BASE_URL` from `config/constants.ts`.
- `fixtures/apiAuthFixture.ts`: exposes `test` and `expect`. Fixtures `authenticatedClientUser` and `authenticatedAdminUser` call `createAndAuthenticateUser` (in `fixtures/helpers/authHelper.ts`) to sign up + log in users and return their tokens.
- `generators/*.ts`: encapsulate faker usage. `generateRandomProduct`, `generateRandomClientUser`, etc. return valid DTOs defined in `types/**`.
- `types/**`: DTO definitions (`ProductCreateDto`, `CartDto`, etc.) keep the tests type-safe.
- `package.json` scripts: `npm run test:api` executes only the API suite via `playwright test tests/api/`.

## HTTP Client Pattern
For every new endpoint:
1. Create a request helper in `http/<domain>/<action>Request.ts`.
2. Export a function per auth flavor (e.g. main call + `WithoutAuth` helper when the tests need to assert 401).
3. Accept the `APIRequestContext`, tokens, ids, and DTOs as typed parameters.
4. Set headers explicitly (`Authorization`, `Content-Type`) and avoid repeating base URLs inside tests.

This keeps tests declarative: specs only orchestrate flows and expectations; request details live in one place.

## Test Structure
Each spec file typically looks like:

```ts
import { test, expect } from '../../../fixtures/apiAuthFixture';
import type { ProductCreateDto, ProductDto } from '../../../types/product';
import { createProduct } from '../../../http/products/createProductRequest';
import { generateRandomProduct } from '../../../generators/productGenerator';

test.describe('POST /api/products API tests', () => {
  test('should create product as admin - 201', async ({ request, authenticatedAdminUser }) => {
    // given
    const { token } = authenticatedAdminUser;
    const payload: ProductCreateDto = generateRandomProduct();

    // when
    const response = await createProduct(request, token, payload);

    // then
    expect(response.status()).toBe(201);
    const body: ProductDto = await response.json();
    expect(body.name).toBe(payload.name);
  });
});
```

### Checklist for Every Test
- Import `test`/`expect` from the API auth fixture (not from `@playwright/test` directly) so client/admin tokens are available.
- Declare DTO typings for request/response bodies for auto-complete and structural assertions.
- Use the generators to build payloads and keep randomness reproducible (they already normalize values to satisfy backend constraints).
- Prepare upstream data using HTTP helpers. Example: admin token creates a product before a client token exercises cart endpoints.
- Call the dedicated request function inside the `// when` section.
- Assert status codes first, then deserialize the JSON and verify specific fields or invariants (e.g. `totalItems` updates, persisted values match).
- When verifying side effects, chain another request (e.g. fetch updated product after PUT, call `getCart` after `clearCart`).
- Keep negative tests lean: assert the status code (and optionally specific fields on the error payload) without duplicating unit-level validation coverage.

### Authentication Patterns
- Use `authenticatedAdminUser` when admin privileges are required (product management).
- Use `authenticatedClientUser` for customer flows (cart endpoints, viewing products).
- For tests that only assert `401`, use the `WithoutAuth` helper and omit fixtures entirely.
- For invalid token scenarios, construct tokens inline (`const invalidToken = 'invalid.token.here';`) to keep intent explicit.

## Adding a New API Test Suite
1. **Add the request helper** under `http/<resource>/<action>Request.ts` following existing files.
2. **Create the spec** at `tests/api/<resource>/<action>.api.spec.ts` with a `test.describe('VERB /path API tests', ...)` header.
3. **Implement scenarios** ordered by HTTP status.
4. **Use generators and fixtures** for setup. Register temporary users/products via API helpers; avoid hard-coded IDs except for negative cases.
5. **Keep tests isolated** by creating and cleaning data via API calls rather than reusing shared state.
6. **Run `npm run test:api` (or a focused `playwright test tests/api/<resource>/...`) headlessly**, ensure green before submitting.

## Useful References
- API definitions are available in `api-docs.json` if an endpoint contract is unclear.
- Existing suites for `products` and `cart` show how to chain helper calls, assert response bodies, and cover auth failures.
- If you suspect a backend bug, keep the failing assertion, capture the response (`const body = await response.json();`), and document the issue instead of deleting the test.

Following this approach ensures new API coverage slots cleanly into the suite and remains consumable for both humans and AI tooling.
